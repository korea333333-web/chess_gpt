## 1. 프로젝트 비전

흔한 웹 체스 게임은 평면적이고 밋밋하다. GAMBIT은 **"진짜 체스판 앞에 앉은 느낌"**을 브라우저에서 구현한다.

- 위에서 살짝 비스듬히 내려다보는 입체 시점으로, 말 하나하나가 나뭇결·대리석·광택까지 살아있는 클래식 스탠턴(Staunton) 스타일.
- 혼자서는 컴퓨터(AI)와, 옆 사람과는 한 화면에서 번갈아, 나중에는 멀리 있는 친구와 온라인으로.
- "체스를 둔다"보다 **"멋진 체스를 둔다"**는 경험에 집중한다.

핵심 차별점: **비주얼 퀄리티**. 게임 규칙은 검증된 라이브러리로 정확하게 처리하고, 우리의 노력은 "보는 순간 와 소리 나는" 3D 연출에 쏟는다.

## 2. 핵심 기능 목록

### 2-1. 입체 체스판 & 카메라
- 3D 체스판을 버드아이뷰(비스듬한 탑다운)로 표시
- 마우스 드래그로 시점을 살짝 회전 / 휠로 줌 인·아웃 (과하지 않게 제한)
- 내 진영이 항상 화면 앞쪽에 오도록 흑/백 시점 자동 전환
- "카메라 리셋" 버튼으로 기본 시점 복귀

### 2-2. 말 & 비주얼 (이 프로젝트의 핵심)
- 클래식 스탠턴 스타일 6종 기물(폰·룩·나이트·비숍·퀸·킹), 흑·백
- PBR 재질: 나뭇결/대리석/광택 금속, 사실적인 반사·음영
- HDRI 환경 조명으로 말 표면에 주변 빛이 비치는 실사감
- 부드러운 그림자(soft shadow), 화면 후처리(앰비언트 오클루전·톤매핑)
- 말 이동 애니메이션(들어서 옮기거나 미끄러지듯), 잡힐 때 사라지는 연출
- 말을 놓는 "딱" 소리 등 효과음

### 2-3. 게임 규칙 엔진 (정확성)
- 합법적인 수만 허용, 선택한 말의 이동 가능 칸 하이라이트
- 체크 / 체크메이트 / 스테일메이트 / 무승부 판정
- 특수 규칙: 캐슬링, 앙파상, 폰 승급(프로모션 — 퀸/룩/비숍/나이트 선택)
- 무르기(undo), 기보 기록 및 다시보기(PGN)

### 2-4. 대전 모드 (1단계)
- **로컬 2인 (핫시트):** 한 기기에서 두 사람이 번갈아 둠
- **vs 컴퓨터 (AI):** 난이도 선택(초급/중급/고급), 강력한 오픈소스 엔진 사용
- 모드 선택 화면 → 게임 시작 → 결과 화면(승패, 다시하기)

### 2-5. 사용자 편의
- 마지막에 둔 수 강조, 체크 상태 시각 경고
- 잡은 말 목록 표시, 양측 기물 점수 차이
- 새 게임 / 항복 / 무르기 / 음소거 버튼

### 2-6. 온라인 대전 (2단계 — 추후 확장)
- 로그인(카카오 / 구글)
- 방 코드로 친구 초대 or 자동 매칭
- 실시간 수 동기화(상대가 둔 수가 즉시 반영), 연결 끊김 처리
- 관전 모드, 채팅(선택)

## 4. 기술적 의사결정

### 4-1. 기술 스택
| 항목 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | Next.js 14 (App Router) | 익숙한 스택, Vercel 자동배포, 온라인 확장 시 서버 일원화 |
| 3D 렌더링 | React Three Fiber + drei + three.js | React로 3D를 선언적으로 다룸. 사실상 업계 표준 |
| 체스 규칙 | chess.js | 이동·체크·캐슬링·앙파상·승급까지 검증된 처리 (직접 안 짬) |
| 컴퓨터 AI | Stockfish (WebAssembly) | 세계 최강 오픈소스 체스 엔진. 난이도/사고시간 조절 가능 |
| 3D 에셋 | 절차적 생성(lathe) + 일부 GLTF + PBR 재질 + HDRI | 실사감과 라이선스 안전 동시 확보 (4-2 참고) |
| 스타일링 | Tailwind CSS | 빠른 UI 구성 |
| 백엔드(2단계) | Supabase (Realtime / Auth / DB) | 이미 보유. 실시간 수 동기화에 적합 |
| 배포 | Vercel (GitHub 연동 자동배포) | 기존 워크플로우 그대로 |
| 버전관리 | GitHub (프라이빗 리포) | |

> 용어 설명 — **PBR**: 빛을 물리적으로 계산해 재질을 사실적으로 표현하는 방식. **HDRI**: 실제 공간을 360°로 담은 조명용 이미지로, 말 표면에 진짜 같은 반사를 만들어 줌. **WebAssembly**: 무거운 프로그램(체스 엔진)을 브라우저에서 빠르게 돌리는 기술.

### 4-2. 말 디자인 방식 (실사감 전략)
"거의 실사"를 라이선스 문제 없이, 모든 기기에서 부드럽게 돌리기 위한 절충안:
- **폰·룩·비숍·퀸·킹:** 옆모습 윤곽선을 회전시켜 만드는 방식(lathe)으로 우아한 "깎은 나무" 형태를 직접 생성 → 라이선스 자유, 용량 작음
- **나이트(말 머리):** 곡면이 복잡해 절차적 생성이 어려움 → 고품질 무료 라이선스(CC0) GLTF 모델 사용하거나 직접 모델링
- 모든 말에 PBR 재질(원목/대리석)과 HDRI 반사를 입혀 통일된 실사감
- 1차 목표 비주얼은 "고급 원목 체스 세트를 좋은 조명에서 촬영한 제품 사진" 수준

### 4-3. 라이선스 / 비용
| 항목 | 라이선스 / 비용 |
|------|----------------|
| three.js / R3F / drei | MIT, 무료 |
| chess.js | BSD, 무료 |
| Stockfish | **GPL**, 무료 (별도 WASM으로 포함, 배포 시 GPL 고지 준수) |
| HDRI / 텍스처 | Poly Haven 등 **CC0**(저작권 자유) 사용 |
| Tailwind / Next.js | MIT, 무료 |
| Vercel / Supabase | 무료 티어 내 운영 가능 |
→ **1단계 외부 비용 0원.** 트래픽이 커지면 Vercel/Supabase 유료 전환 검토.

### 4-4. 성능 목표
| 항목 | 목표 |
|------|------|
| 데스크톱 프레임 | 60fps 유지 |
| 중급 사양 모바일 | 30fps 이상 |
| 초기 로딩 | 3초 이내 (3D 에셋 지연 로딩) |
| AI 응답 시간 | 난이도별 0.1초 ~ 2초 |
| 합법 수 계산·하이라이트 | 즉시(체감 지연 없음) |

### 4-5. 온라인 실시간 대전을 고려한 설계 (중요)
온라인 대전은 2단계지만, **1단계부터 그걸 염두에 두고 만든다.** 나중에 큰 재작성 없이 붙이기 위해서.
- **게임의 "상태"와 "화면"을 분리:** 체스판의 현재 상황(어떤 말이 어디 있는지)을 한 곳에서 관리하고, 3D 화면은 그 상태를 "보여주기만" 하게 만든다.
- 그러면 2단계에서 상대의 수가 인터넷으로 들어와도, **그냥 같은 "상태"를 바꾸기만** 하면 화면이 알아서 따라온다 → 핫시트·AI·온라인이 같은 구조를 공유.
- 한 수의 표현을 표준 형식(이동 좌표 + FEN/PGN)으로 통일 → 서버로 주고받기 쉬움.
- 결론: **로컬에서 사람끼리 / 컴퓨터와 두는 것이나, 온라인에서 두는 것이나 내부 구조는 동일.** 1단계를 제대로 만들면 온라인은 "연결선"만 추가하는 일이 된다.

## 5. 수익 모델 (선택)
- **1단계: 완전 무료.** 먼저 잘 만들고 사람을 모으는 데 집중.
- 추후 옵션(가벼운 수준):
  - 말/보드 테마 스킨(원목·대리석·유리·메탈 등) 일부 유료
  - 비침습적 광고
  - 고급 분석(내 대국 복기, 실수 짚어주기)
- 초기에는 수익보다 **완성도·비주얼 임팩트** 우선.

## 6. 대상 사용자
- 체스를 두고 싶은 일반 사용자(캐주얼 ~ 중급)
- "예쁘고 고급스러운" 체스 경험을 원하는 사람
- 1차로는 **데모/콘텐츠/포트폴리오** 용도로도 강력 (보는 순간 임팩트)
- 추후 온라인 대전으로 친구·지인과 함께 즐기는 사용자
